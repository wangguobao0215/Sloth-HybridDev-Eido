# Skill-G: GTM 知识架构师 (WisdomCore-GTMArchitect)

> 本文档为 Skill-G 的完整参考手册，涵盖 AI 能力脱敏翻译规则、五大铁律、
> 销售作战卡模板、发版说明组装、Feature Brief、审签流程及翻译质量保障。

---

## 1. Skill 概况

| 属性 | 内容 |
|------|------|
| 名称 | **WisdomCore-GTMArchitect** |
| Skill ID | `wisdomcore-g` |
| 目标用户 | 产品市场经理（PMM）、销售团队、市场运营、内容团队 |
| 核心价值 | 将技术语言安全地翻译为市场语言——"脱敏"但不"脱真"，让销售能自信地讲 AI 故事 |
| 输入 | PRD 文档、协作协议、实验报告、Bad Case Registry |
| 输出 | 销售作战卡、发版说明、Feature Brief、翻译映射表 |
| 依赖 | 所有文档目录、SQLite 索引、Skill-H 健康度数据 |
| 部署阶段 | Phase 2（与 Skill-H 同期部署，作为数据消费者） |

**设计哲学**：混合研发最危险的"最后一公里"是技术到市场的翻译。翻译过度会变成
虚假宣传（"AI 准确率高达 99%"），翻译不足会丧失竞争力（"我们有个模型，准确率
80%"）。核心原则是**"脱敏但不脱真"**——用客户听得懂的语言讲真实的技术故事，
所有量化表述都可追溯到实验数据。

## 2. AI 能力"脱敏"翻译规则表

### 2.1 翻译规则总表

| 技术表述 | 市场表述 | 适用场景 | 注意事项 |
|---------|---------|---------|---------|
| "置信度 < 0.6 时触发降级" | "系统具备智能兜底机制，确保服务连续性" | 销售话术、产品白皮书 | 不得省略"兜底"含义 |
| "MRR@10 = 0.85" | "搜索结果精准度达到行业领先水平" | 产品介绍、官网 | 需注明"基于内部测试集" |
| "BERT-base 微调" | "采用深度学习技术，基于海量数据训练" | 通用描述、媒体稿 | 不需要提及具体模型名 |
| "P99 延迟 180ms" | "毫秒级响应速度" | 性能宣传、销售材料 | 仅当 P99 < 500ms 时可用 |
| "模型准确率 80%" | "AI 辅助决策准确率高达 80%" | 客户沟通、提案 | 必须加"辅助"，不可暗示 100% 自动化 |
| "F1-Score = 0.78" | "在准确性和覆盖率之间取得优秀平衡" | 技术方案介绍 | 不适用于非技术受众 |
| "Recall = 0.92" | "极少遗漏，覆盖超过 90% 的目标场景" | 产品亮点、功能介绍 | 必须说明覆盖的是什么 |
| "混合降级策略" | "多层保障机制，确保服务永不中断" | 安全性宣传 | "永不中断"可改为"高可靠" |
| "A/B 测试提升 15%" | "经过严格对比测试，效果提升显著" | 案例研究 | 可选择是否披露具体数值 |
| "训练数据 100 万条" | "基于百万级真实业务数据训练" | 规模宣传 | 必须是真实数据量 |

### 2.2 翻译执行流程

```
读取 CA 文档（能力描述 + 降级策略）
  -> 读取关联 EXP 文档（实验数据 + Bad Case）
  -> 应用翻译规则引擎
  -> 生成市场化表述候选（2-3 个版本）
  -> 事实核查（所有数值追溯到实验数据）
  -> 禁用词扫描（"绝对"、"100%"、"永远不会出错"等）
  -> 输出翻译结果 + 数据溯源注释
```

## 3. 五大翻译铁律

### 铁律一：绝不过度承诺

准确率 80% 绝对不能说"高达 95%"。翻译可以美化表述方式但不能扭曲数值。
Agent 会对所有数值型表述进行事实核查。

### 铁律二：必须提及降级

任何对外材料中涉及 AI 功能时，必须用适当方式提及系统的可靠性保障。
技术侧的"降级策略"在市场侧应表述为"智能保障机制"或"多层可靠性设计"。

### 铁律三：量化表述可追溯

所有量化数据（"准确率 80%"、"毫秒级响应"）必须能追溯到具体实验报告。
Agent 会在 GTM 文档中添加内部注释：
```html
<!-- 数据来源：EXP-2026-005, MRR@10=0.82 -->
```

### 铁律四：市场稿须经审批

所有生成的 GTM 资产默认为 `Draft` 状态，必须经过 Product Owner 审签后
才能标记为 `Approved`。

### 铁律五：一致性保障

同一功能在不同 GTM 资产（销售卡、发版说明、Feature Brief）中的描述必须一致。
Agent 会进行交叉检查。

## 4. 销售作战卡模板（Sales Battle Card）

### 4.1 Frontmatter

```yaml
---
id: SBC-2026-001
title: "销售作战卡: {功能名称}"
type: sales_battle_card
status: Draft
created_at: YYYY-MM-DD
updated_at: YYYY-MM-DD
owner: "{产品市场经理}"
linked_agreements: [CA-YYYY-NNN]
linked_experiments: [EXP-YYYY-NNN]
linked_prd: PRD-YYYY-NNN
target_release: vX.Y.Z
review_chain: ["PMM", "产品经理", "法务(optional)"]
---
```

### 4.2 正文结构

#### 一句话卖点
> 用一句直击痛点的话概括产品价值

#### 目标客户画像

| 维度 | 描述 |
|------|------|
| 行业 | 适用行业 |
| 规模 | 客户规模门槛 |
| 痛点 | 核心业务痛点 |
| 决策者 | 购买决策人角色 |
| 使用者 | 实际使用人群 |

#### 功能亮点
功能编号、亮点名称、一句话解释、客户价值。

#### 技术底气（内部参考，不对外展示）
- 实验数据引用（指标值 + 实验 ID）
- 降级策略说明（L1/L2/L3 分级）
- 已知限制（关联 Bad Case ID）
- 数据来源（CA + EXP ID）

#### 竞品差异
按维度对比：语义理解深度、响应速度、降级保障、个性化能力、部署方式等。

#### 常见异议处理

| 客户异议 | 应对策略 | 支撑证据 |
|---------|---------|---------|
| "AI 不靠谱" | 强调测试验证 + 保障机制 | CA 降级策略 + EXP 指标 |
| "太贵了" | 量化 ROI（零结果率降低 -> 转化率提升） | PRD 业务分析 |
| "接入复杂" | 强调标准 API + 快速接入 | 技术接入文档 |

#### 演示要点
- **必须展示的场景**：核心能力 + 降级无感切换
- **绝对不能演示的场景**：Bad Case 风险场景（多义词、能力边界外等）

## 5. 发版说明自动组装（Release Notes Auto-Assembly）

### 5.1 组装流程

```
用户指定版本号（如 v2.1.0）
  -> Step 1: 扫描 PRDs，筛选 target_release 匹配的需求
  -> Step 2: 查找每个 PRD 关联的 CA 和 EXP
  -> Step 3: 分类：新功能 / 智能升级 / 性能优化 / 已知问题
  -> Step 4: 从 Bad Case Registry 拉取 open + high 的 Bad Cases
  -> Step 5: 从 CA 中拉取当前生效的降级策略
  -> Step 6: 应用翻译规则引擎转化为市场语言
  -> Step 7: 组装为标准发版说明格式
  -> Step 8: 事实核查 + 禁用词扫描
  -> Step 9: 输出 Draft 版本供审签
```

### 5.2 分类逻辑

| 分类 | 判断条件 |
|------|---------|
| 新功能 | PRD 为新建且关联 CA 为首次创建 |
| 智能升级 | 已有功能的 AI 能力提升（CA 更新且关联新 EXP） |
| 性能优化 | EXP 主要指标为延迟/吞吐量改善 |
| 已知问题 | Bad Case Registry (severity=high, status=open) + 活跃降级策略 |

### 5.3 输出模板 Frontmatter

```yaml
---
id: RN-2026-vX.Y.Z
title: "Release Notes vX.Y.Z"
type: release_notes
status: Draft
created_at: YYYY-MM-DD
owner: "{产品市场经理}"
linked_prds: [PRD-YYYY-NNN, ...]
linked_agreements: [CA-YYYY-NNN, ...]
linked_experiments: [EXP-YYYY-NNN, ...]
---
```

正文包含：发布日期、新功能概览（含翻译后的市场语言 + 内部溯源注释）、
智能升级、性能优化、已知问题与限制表、致谢。

## 6. Feature Brief 模板（One-Pager）

### 6.1 生成逻辑

```
指定功能名称或 CA ID
  -> 读取 PRD（问题定义 + 用户需求）
  -> 读取 CA（解决方案定义 + 验收标准 + 降级策略）
  -> 读取关联 EXP（实验结果 + 关键指标）
  -> 读取 Bad Cases（已知限制）
  -> 应用翻译引擎（技术 -> 市场）
  -> 生成 One-Pager
```

### 6.2 模板结构

```yaml
---
id: FB-2026-NNN
title: "Feature Brief: {功能名称}"
type: feature_brief
status: Draft
linked_agreement: CA-YYYY-NNN
---
```

正文包含：
- **问题陈述** — 来源 PRD 的用户调研与数据分析
- **解决方案** — 来源 CA 协作协议的能力定义
- **关键指标** — 翻译后的市场化表述 + 内部溯源注释
- **已知限制与路线图** — 当前限制 + 影响程度 + 解决计划
- **客户价值** — 核心价值主张（3-5 条）

## 7. GTM 审签流程（Review & Approval Workflow）

### 7.1 状态流转

```
Draft（AI 生成初稿）
  -> UnderReview（提交审签）
  -> [审签意见]
    -> 若通过 -> Approved（审签通过） -> Published（正式发布）
    -> 若驳回 -> Draft（返回修改）
```

### 7.2 审签链配置

| 资产类型 | 必须审签人 | 可选审签人 | SLA |
|---------|-----------|-----------|-----|
| 销售作战卡 | Product Owner | 销售总监 | 3 个工作日 |
| 发版说明 | Product Owner + 技术负责人 | Marketing | 5 个工作日 |
| Feature Brief | Product Owner | Marketing, 法务 | 3 个工作日 |
| 对外宣传稿 | Product Owner + Marketing + 法务 | CEO | 7 个工作日 |

### 7.3 审签记录格式

```yaml
review_history:
  - reviewer: "{审签人（角色）}"
    action: approved | revision_requested
    date: YYYY-MM-DD
    comment: "审签意见"
```

### 7.4 版本管理

- 每个 GTM 资产都有版本号（`v1.0`, `v1.1`, `v2.0`）
- 小改动（措辞调整）-> 小版本号递增
- 大改动（功能更新、指标变化）-> 大版本号递增
- Published 状态的资产被修改时，自动创建新版本而非覆盖

> **注**：GTM 资产的版本历史通过 Git 提交记录进行追溯管理。文档 Frontmatter 中的
> `version` 字段用于标记当前版本号（如 `v1.0`），便于审签和引用时快速识别版本。

## 8. 翻译质量保障（Translation Quality Assurance）— 三层检查

### 8.1 第一层：禁用词扫描（Rule-Based Pre-Check）

禁用词清单：

| 类别 | 禁用词 | 原因 |
|------|--------|------|
| 绝对化用语 | "100%准确"、"永远不会出错"、"完美"、"绝对" | 不得做绝对承诺 |
| 极限用语 | "最好的"、"第一"、"唯一" | 涉嫌广告法违规 |
| 过度承诺 | "取代人工"、"全自动"、"无需人工" | 应表述为"辅助"/"减少" |
| 未验证表述 | "业界首创"、"遥遥领先" | 需提供证据支撑 |

### 8.2 第二层：事实核查（Fact-Check）

- 提取所有量化表述（"准确率 80%"、"毫秒级响应"、"提升 15%"）
- 逐一与关联实验报告的 `key_metrics` 比对
- 如表述与实验数据不符，输出警告
- 如表述找不到实验数据支撑，要求补充数据来源或删除该表述

### 8.3 第三层：一致性检查（Consistency Check）

- 查询同一功能关联的所有 GTM 资产（销售卡、发版说明、Feature Brief）
- 提取各资产中对该功能的描述
- 计算语义相似度，若相似度 < 0.7 则标记为不一致
- 输出不一致详情及统一表述建议

### 8.4 质量检查输出示例

```markdown
## GTM 质量检查报告

### 禁用词扫描
通过 -- 未发现禁用词

### 事实核查
通过 -- 3 项量化表述均已追溯到实验数据
  - "搜索精准度达到行业领先水平" <- EXP-2026-005, MRR@10=0.82
  - "毫秒级响应速度" <- EXP-2026-005, P99=280ms
  - "覆盖超过 90% 的目标场景" <- EXP-2026-005, Recall=0.92

### 一致性检查
发现 1 处不一致：
  - 销售作战卡描述为"四级自动降级"，发版说明描述为"多层保障机制"
  - 建议统一表述为"多层智能保障机制"
```

## 10. 与其他 Skill 的交互

| 交互方向 | 交互内容 | 数据流 | 触发时机 |
|---------|---------|--------|---------|
| G <- 父 Skill | 读取所有文档目录（PRD, CA, EXP） | 文件读取 | GTM 资产生成时 |
| G <- Skill-S | 读取协作协议（能力描述、降级策略、验收标准） | 读取 CA 文件 | 翻译引擎执行时 |
| G <- Skill-A | 读取实验报告 + Bad Case Registry | EXP 文件 + SQLite | 事实核查 + 已知问题 |
| G <- Skill-H | 获取健康度数据、文档状态摘要 | SQLite 查询 | 发版说明组装时 |
| G -> 父 Skill | 写入 GTM 资产到 `04_GTM_Knowledge/` | 文件写入 + SQL | 资产生成完成时 |
| G -> 市场/销售 | 输出最终 GTM 资产供外部使用 | Markdown 文件 | 状态变为 Published |

**数据消费优先级**：
1. **协作协议 (CA)** — 最重要的输入源，提供能力定义和降级策略
2. **实验报告 (EXP)** — 提供量化数据支撑和 Bad Case 信息
3. **产品需求 (PRD)** — 提供业务背景和竞品分析
4. **健康度数据 (Skill-H)** — 提供系统整体状态参考
